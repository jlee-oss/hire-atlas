# 일반화 가드레일

이 문서는 `Hire Atlas`의 추출 품질을 올리는 과정에서 `데이터셋 맞춤 보정`이 과해져 오버피팅으로 기울지 않도록 막는 운영 기준을 정리합니다.

## 1. 왜 필요한가

- 현재 데이터만 보고 규칙을 계속 덧대면, 지금 보이는 오류는 줄어들 수 있어도 다음에 유입되는 공고에서 더 쉽게 무너집니다.
- 특히 `대표 라벨 재선정`, `그룹 보정`, `표시용 summary 보정`은 잘 쓰면 유용하지만, 특정 회사/특정 표현에 맞춰지기 쉬운 영역입니다.
- 따라서 앞으로는 `지금 예뻐 보이는 결과`보다 `새 데이터에서도 유지되는 신뢰도`를 더 우선합니다.

## 2. 남겨도 되는 규칙

아래는 일반화 가능성이 높고 계속 유지해도 되는 규칙입니다.

- 조사/접속어/학력/경력/포털 문구 제거
  - 예: `위한`, `또는`, `학력`, `석사`, `제품`, `서비스`
- 표기 정규화
  - 예: `엘엘엠 -> LLM`, `파이토치 -> PyTorch`, `오엔엔엑스 -> ONNX`
- 빈 값, 중복 값, title echo 제거
- `low confidence`일 때 비우는 전략
- 너무 긴 keyword, 문장형 keyword 제거
- 요약문 종결형 제거 같은 표현 정리

이 규칙들은 특정 회사나 특정 날짜에 의존하지 않기 때문에 일반화 안전도가 높습니다.

## 3. 줄여야 하는 규칙

아래는 효과는 있을 수 있지만 오버피팅 위험이 높은 규칙입니다.

- 특정 기술명을 과도하게 우선하는 라벨 선택
  - 예: `PyTorch`, `TensorFlow`, `SQL`만 보고 대표 라벨을 결정
- 특정 도메인 힌트를 강하게 재배치하는 보정
  - 예: summary에 특정 단어가 있으면 무조건 대표 라벨 교체
- 회사/직무별로 보정 방향이 사실상 고정되는 규칙
- 보드 단계에서 의미를 재해석하는 과도한 후처리
- `현재 데이터셋에서 자주 보였다`는 이유만으로 넣는 예외 처리

이 영역은 원칙적으로 `임시 보정`으로 간주하고, 유지 여부를 별도 검증으로 통과해야만 남깁니다.

## 4. 앞으로의 기본 원칙

### 4-1. 추출과 그룹화를 분리

- 모델은 먼저 `중립적인 구조화 신호`를 뽑습니다.
- 그룹은 이 신호를 조합해 만듭니다.
- 즉 모델이 처음부터 `예쁜 그룹 제목`을 만들려고 하지 않게 합니다.

추천 추출 축:

- `domain_signal`
- `model_signal`
- `system_signal`
- `data_signal`
- `workflow_signal`
- `confidence`

### 4-2. 모르면 비운다

- 억지로 대표 라벨을 채우는 것보다 `low`로 남기는 편이 낫습니다.
- 서비스 신뢰도는 `빈 값`보다 `잘못된 확신`에 더 크게 손상됩니다.

### 4-3. 보드 단계 보정은 최소화

- 보드에서 허용되는 후처리는 `표기 통일`, `길이 제한`, `노이즈 제거`까지를 기본으로 합니다.
- 의미론적 재해석은 가능한 한 모델 출력과 schema 쪽으로 이동합니다.

## 5. 검증 체계

앞으로 검증은 두 축으로 운영합니다.

### 5-1. 고정 평가셋

- 기존 `data/eval_set.json`
- 역할/군집/품질 편차를 섞은 기준셋
- prompt 변경의 기본 비교 기준

### 5-2. 증분 홀드아웃

- 가장 최근에 들어온 공고를 따로 고정
- 이 셋은 `튜닝에 직접 쓰지 않는` 검증셋으로 유지
- 새 prompt나 후처리를 적용할 때 이 셋에서 악화되면 보류

현재 추천:

- 최근 유입 공고 상위 48건을 `incremental holdout`으로 사용
- 나머지는 core set으로 유지

## 6. 채택 기준

새 규칙/새 prompt는 아래를 만족해야 채택합니다.

1. 고정 평가셋에서 개선 또는 최소 유지
2. 증분 홀드아웃에서 악화 없음
3. 금지어 발생률 증가 없음
4. `low` 비율 급증 없음
5. 보드 보정 없이도 읽을 수 있는 값이 늘어남

## 7. 현재 판단

- `field_aware_v3`는 현재 기준에서 채택 가능
- 다만 board 단계의 일부 focus 재선정 규칙은 `임시 보정`으로 간주
- 따라서 다음 단계는:
  - 규칙을 더 추가하는 것보다
  - `증분 홀드아웃`을 기준으로 유지/제거를 결정하는 것

## 8. 다음 운영 방식

앞으로는 아래 순서로 진행합니다.

1. 새 데이터 동기화
2. `incremental holdout` 재생성
3. prompt 또는 후처리 변경
4. `core eval + incremental holdout` 동시 비교
5. 둘 다 통과한 경우에만 보드 반영

이 문서는 `현재 데이터에 맞춘 미세한 땜질`보다 `증분 데이터에서도 무너지지 않는 구조`를 우선한다는 기준 문서로 유지합니다.
