# 모델 튜닝 판단 우선순위

이 문서는 `추가 러닝`이나 `파인튜닝`을 언제 검토해야 하는지, 그 전에 무엇을 먼저 확인해야 하는지를 우선순위 순서대로 정리합니다.

## 1. 가장 먼저 확인할 것

### 1-1. 지금 정확도가 실제로 측정되고 있는가

아래가 없으면 아직 `정확도`를 논할 단계가 아닙니다.

- 고정 평가셋 `data/eval_set.json`
- 증분 홀드아웃 `data/incremental_eval_set.json`
- 사람이 직접 검토한 골드 라벨 일부

없다면 먼저 평가 체계를 만든 뒤에 모델 변경을 논의합니다.

### 1-2. 현재 개선이 일반화되고 있는가

다음 조건이 같이 보장되어야 합니다.

- core 평가셋에서 baseline보다 나아짐
- incremental holdout에서 baseline보다 나아짐
- 금지어/문장형 keyword가 늘지 않음

하나라도 깨지면 튜닝보다 먼저 prompt/후처리 회귀를 막아야 합니다.

## 2. 그다음 확인할 것

### 2-1. 더 강한 모델 비교를 했는가

현재 모델 하나만 보고 `파인튜닝이 필요하다`고 결론 내리면 안 됩니다.

먼저 같은 benchmark suite로:

- 현재 모델
- 더 강한 범용 모델

을 비교해야 합니다.

그 이유는 단순합니다.

- 더 강한 모델이 큰 폭으로 좋아지면
  - 지금 문제는 `학습 부족`보다 `모델 용량 한계`일 수 있습니다.
- 더 강한 모델도 큰 차이가 없으면
  - 그때는 `schema`, `라벨`, `튜닝` 쪽이 더 중요해집니다.

## 3. 그다음 확인할 것

### 3-1. 실패가 반복되는 구조적 오류인가

아래처럼 `같은 유형`이 반복돼야 튜닝 명분이 생깁니다.

- 희소 도메인에서 반복적으로 low
- focusLabel이 반복적으로 너무 넓음
- summary가 계속 role/title echo로 흐름
- 같은 업무 유형을 항상 특정 기술명으로 잘못 대표함

반대로 실패가 산발적이면 튜닝보다 라벨 정리나 후처리 조정이 먼저입니다.

## 4. 튜닝 이전 마지막 확인

### 4-1. 학습 대상으로 쓸 정답셋이 충분한가

아래가 없으면 파인튜닝을 해도 불안정합니다.

- 최소 50~100건 이상 검수된 core 라벨
- 20~30건 이상 incremental 라벨
- `summary / focusLabel / keywords` 또는 구조화 signal의 정답 정의

즉, `무엇을 맞다고 볼 것인지`가 먼저 고정되어야 합니다.

## 5. 실제 우선순위

현재 프로젝트에서 모델 관련 의사결정 우선순위는 아래와 같습니다.

1. `평가 체계 확인`
2. `core + incremental 일반화 확인`
3. `더 강한 모델 비교`
4. `반복 오류 유형 확인`
5. `골드셋 확보`
6. `그 다음에만 파인튜닝 판단`

## 6. 언제 튜닝으로 넘어가는가

아래를 모두 만족하면 파인튜닝 단계로 넘어갑니다.

- core + incremental 둘 다에서 현재 최적 prompt가 안정적
- 더 강한 모델 비교 완료
- 그래도 반복 오류가 남음
- 골드셋이 준비됨
- 튜닝 목표가 `자유 생성`이 아니라 `구조화 추출`로 명확함

## 7. 현재 판단 원칙

현 시점에서는 아래가 기본 원칙입니다.

- `프롬프트 + 입력 설계 + 최소 후처리`가 먼저
- `더 강한 모델 비교`가 다음
- `파인튜닝`은 그 이후

즉, 파인튜닝은 `지금 바로 해야 하는 기본 단계`가 아니라, 앞선 검증을 통과한 뒤에만 들어가는 고비용 단계로 본다.
