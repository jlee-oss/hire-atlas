# 모델 품질 원칙

- 작성일: `2026-04-10`
- 상태: `현재 작업의 최우선 기준`

## 왜 이 문서가 필요한가

- 현재 공고 분류, 키워드 추출, 군집화 품질은 이후 기능의 기반이다.
- 이 프로젝트는 차후 `교육 과정/단원 -> 카드 -> 공고`를 연결하는 graph RAG 체계로 확장될 예정이다.
- 따라서 지금의 오분류, 잘못된 키워드, 잘못된 군집은 단순 화면 오류가 아니라 미래 그래프 엣지 오염으로 이어진다.
- 이 문서는 `숫자를 좋아 보이게 만드는 임시 대응`이 아니라 `모델 성능 자체를 높이는 방향`을 강제하기 위해 기록한다.

## 고정 원칙

### 1. 최우선 목표는 현재 품질 자체의 개선이다

- 최우선은 `현재 분류 정확도`, `키워드 추출 정확도`, `군집 품질`이다.
- 다음 단계 기능을 서두르기 전에 현재 모델 결과가 서비스 가능한 수준인지 먼저 본다.
- 지금 품질이 낮으면 이후 graph RAG, 교육 과정 매핑, 카드-공고 연결 품질도 함께 무너진다.

### 2. 배제나 가리기로 품질 문제를 숨기지 않는다

- `exclude`, `hide`, `fallback`, `blacklist`는 응급처치일 수는 있어도 본질 해결로 간주하지 않는다.
- 공고 수를 줄여서 숫자만 안정화하는 방식은 채택하지 않는다.
- 문제가 생겼을 때 기본 접근은 `모델/입력/출력/평가` 개선이지 `보이지 않게 만들기`가 아니다.

### 3. raw 채용문장이 카드 라벨을 직접 만들면 안 된다

- `requirements`, `preferred`, `detailBody` 원문이 카드 제목이나 군집 seed를 직접 만들면 안 된다.
- 카드/군집/그래프의 핵심 입력은 `focusLabel`, `highlightKeywords`, `structuredSignals` 같은 모델 기반 구조화 신호여야 한다.
- raw 문장은 증거 문장과 참고 문장으로는 쓸 수 있지만, 핵심 분류 기준으로 쓰지 않는다.

### 4. blacklist는 임시 응급처치일 뿐이다

- `이상이신`, `채용절차법`, `경력무관`, `계약직`, `광주지사` 같은 이상 표현을 하나씩 막는 방식은 임시 대응으로만 사용한다.
- 같은 종류의 문제가 반복되면 `규칙 추가`가 아니라 `입력 경로와 모델 출력 체계`를 다시 본다.
- 궁극적으로는 `나쁜 단어를 많이 아는 시스템`이 아니라 `좋은 신호만 구조적으로 남기는 시스템`을 만든다.

### 5. 모델이 다시 읽고 다시 분류하는 체계가 우선이다

- 새 데이터가 들어올 때마다 `요약`, `직무 분류`, `키워드`, `structuredSignals`, `군집 입력`이 다시 계산되어야 한다.
- 데이터가 바뀌어도 같은 문제가 반복되지 않게 하려면 룰보다 `모델 재판정 + 게이트`가 우선이다.
- 휴리스틱은 안전장치로만 둔다.

### 5-1. guard 통과를 모델 개선으로 기록하지 않는다

- `model_exclude_should_be_recovered`, `ENGINEER_FOCUS_BLOCKLIST`, focus rebalance 같은 코드는 모델 판단 자체를 개선한 것이 아니라 모델 오판이 데이터 삭제나 focus 왜곡으로 이어지지 않게 막는 안전장치다.
- 이 안전장치로 `optimizationScore=100`이 되더라도 그것은 `runtime 안정화`이지 `model improvement 완료`가 아니다.
- 모델 개선 완료로 기록하려면 guard가 개입한 row를 goldset/review set으로 고정하고, 새 프롬프트/모델이 guard 없이도 같은 기준을 통과해야 한다.
- 따라서 guard는 운영 중단을 막는 응급 구조이고, 이후 작업은 `모델 재판정 -> 기존 판단 대비 -> goldset 반영 -> override 교체` 순서로 이어져야 한다.

### 6. 하네스는 실제 실패와 문맥을 구분해야 한다

- 품질 하네스는 `오분류`와 `문맥 존재`를 같은 실패로 세면 안 된다.
- `focusLabel`이나 첫 keyword가 잘못된 경우와, 같은 row 안에 특정 도메인 문맥이 남아 있는 경우는 분리해서 본다.
- 다음 remediation target은 `dominance failure`와 `focus specificity gap`이어야 한다.
- 하네스가 과민하면 모델 품질이 아니라 지표 정의가 다음 병목이 된다.

## 현재 작업 방향

### 분류

- `roleGroup` 오분류를 줄이는 것을 최우선으로 둔다.
- 직무 경계 케이스는 제거하지 않고 모델이 다시 판정하게 한다.
- 사람이 교정한 사례는 role goldset으로 쌓는다.

### 키워드 추출

- Gemma 기반 추출을 중심으로 전체 체계를 점검한다.
- 절차/학력/경력 문구가 키워드로 올라오지 않게 한다.
- 같은 키워드의 순서만 바뀌거나, 넓은 키워드가 여러 카드에 중복되는 문제를 줄인다.
- 핵심은 `보기 좋은 키워드`가 아니라 `그래프에 써도 되는 신호`를 만드는 것이다.

### 군집화

- 군집 제목은 raw text가 아니라 구조화 신호에서 나온다.
- `경력무관`, `계약직`, `기획자`, `상세채용`, `모집요강` 같은 메타 단어는 군집 핵심어가 되어서는 안 된다.
- 군집은 `무엇을 하는 공고인가` 기준으로 읽혀야 한다.

## 평가 원칙

- 품질 판단은 `보이는 카드 수`보다 `맞는 카드인가`를 우선한다.
- 릴리스 게이트에는 최소한 아래가 포함되어야 한다.
  - role 정확도
  - focus 정확도
  - keyword 품질
  - cluster purity
  - 증분 데이터 일반화
- 새 데이터가 들어왔을 때 같은 수준을 유지하지 못하면 배포 품질로 보지 않는다.

## 금지되는 접근

- 오분류를 발견했을 때 먼저 제거부터 하는 것
- raw 문장을 카드 라벨로 쓰는 것
- blacklist를 계속 덧붙이며 문제를 해결했다고 판단하는 것
- 화면상으로만 덜 보이게 만들어 품질이 올라간 것처럼 해석하는 것

## 실무 해석

- 응급처치는 가능하다. 다만 응급처치를 `해결 완료`로 기록하지 않는다.
- 근본 해결은 항상 아래 질문으로 돌아간다.
  - 이 값은 모델이 낸 구조화 신호인가
  - raw 모집 문장이 카드 체계를 오염시키고 있지 않은가
  - 새 데이터가 들어와도 같은 방식으로 유지되는가
  - 이후 graph RAG에 연결해도 안전한가

## 결론

- 현재 단계의 핵심은 `덜 보여주기`가 아니라 `더 맞게 읽기`다.
- 배제 중심 안정화는 금지한다.
- 앞으로의 품질 개선은 `모델 성능 자체 향상`, `구조화 신호 중심 체계`, `평가셋과 게이트`를 기준으로 진행한다.
